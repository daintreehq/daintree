<decisions_made>

- RFC 8252 compliant: system browser via shell.openExternal, NOT BrowserWindow popup (rejected by maintainer gregpriday in #4563)
- CDP Fetch.requestPaused to fix redirect_uri mismatch at token exchange (RFC 6749 §4.1.3 requires matching redirect_uri). Quorum confirmed this is the only viable approach without modifying user apps.
- CDP reading app webview sessionStorage is NOT an RFC 8252 violation (quorum consensus — RFC prohibits access to IdP page, not app's own webview)
- redirect_uri rewrite requires IdP to accept http://127.0.0.1:\* (RFC 8252 §7.3 recommends this; confirmed working with real Keycloak 26.1)
- No changes to user apps (zero app modifications). All logic is in Canopy.
- Keycloak dev mode needs `--spi-cookie-default-secure=false` and `KC_HEALTH_ENABLED=true` for HTTP testing
- Keycloak 26.1 requires disabling VERIFY_PROFILE required action + setting firstName/lastName on test user
  </decisions_made>

<context>
The architecture is: blocked OAuth redirect in dev-preview → user clicks "Sign in via Browser" → main process starts ephemeral HTTP server on 127.0.0.1:0 → rewrites redirect_uri in auth URL → shell.openExternal opens system browser → user authenticates at IdP → IdP redirects to loopback server → Canopy captures code+state → attaches CDP Fetch.enable to webview → navigates webview to original callback URL → webview JS fires token exchange POST → CDP intercepts and rewrites redirect_uri in POST body → IdP validates matching redirect_uri + PKCE → tokens issued.

The only remaining risk is whether CDP Fetch.requestPaused actually fires on Electron 41 webview webContents. The protocol spec supports it, Canopy already uses CDP for console capture on the same webContents, and the Keycloak test proves the redirect_uri rewrite is necessary and sufficient. The gap is purely "does Electron 41 implement this correctly."
